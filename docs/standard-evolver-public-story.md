# Standard Evolver：让业务标准随着数据进化

很多 AI 使用方式停在“一次回答”：这次分析很精彩，下次又从零开始。我的想法很简单：不要只让 AI 给结论，而是让每一次分析都留下可复用的经验，并推动下一轮方法进化。

更准确地说，它想解决的是一个业务学习问题：AI 不是拿到数据就直接分析，而是先理解用户的业务，先建立一套判断标准，再用用户给的数据不断校准这套标准。

这套方法不依赖任何具体应用形态。核心是业务标准在数据里不断进化；界面、文档、知识库或其他系统只是承载方式。

它更像一个轻量循环：

1. 先说明业务：用户先讲自己做什么、服务谁、要判断什么、什么结果算好。
2. 再建立标准：AI 帮用户整理出业务维度、分类标签、证据规则、优先级和排除项。
3. 再分析数据：用户给数据、案例、素材、对话或记录，AI 按当前标准去分析。
4. 再调整标准：如果数据证明原标准太粗、太偏或漏了关键情况，就只改必要的一小处。
5. 最后沉淀：把新的规则、案例、判断边界和开放问题存下来，下一次分析从新标准开始。

这里的“沉淀”不是把全文保存下来，而是把未来会影响业务判断的东西留下来。比如：哪个维度应该新增，哪个标签定义不清楚，什么证据只能算弱信号，什么情况必须排除，哪些案例可以作为以后判断的样本。

它的重点不是把信息展示得多漂亮，而是让系统越来越懂这门业务怎么判断。一次分析产出一个结论，十次分析校准一套标准，一百次分析以后，就会形成一套带有业务经验的分析方法。

## 它的专业意义

我觉得这套方法可以用五个词概括：

- Trace-based：不只看最终答案，也看中间轨迹。失败尝试、用户纠正、返工原因、命令日志、被拒绝的方案，都是学习材料。
- Procedural memory：沉淀的不是资料本身，而是“以后怎么判断”。例如怎么分类、怎么打分、什么时候该谨慎、什么证据不够强。
- Bounded change：每轮只进化标准里的一个小点。这样下一次才知道到底是哪条规则产生了效果。
- Validation gate：没有验证就不算学会。一个新业务规则必须在下一批数据、复盘或用户反馈里被检验。
- Portable storage：先用普通 Markdown、项目文档、issue、笔记或 skill 文件承载，不急着上数据库和复杂平台。

可以把它做成一个通用 skill：当用户要做研究、产品分析、素材分析、客户分析、项目复盘或任何业务数据分析时，先触发这个 skill，把流程分成三部分：

- 一套“当前业务标准”：这类业务应该看哪些维度、用什么标签、按什么证据判断。
- 一次“基于标准的数据分析”：这批数据说明了什么，哪里符合标准，哪里挑战标准。
- 一段“标准更新记录”：哪些规则被确认、修改、新增或暂时保留。

举个小例子：一次产品分析发现，AI 总是把“用户说喜欢”直接当成“用户愿意付费”。这时不需要做一个复杂系统，只要沉淀一条经验：偏好信号和付费信号要分开。下一轮进化动作可以很小：在产品分析 checklist 里加一条“是否有真实付费、采购、搜索、招聘或投诉等需求信号”。之后再看新分析有没有少犯这个错误。

换成任何业务分析产品也一样：一开始不应该急着处理所有数据，而是先问业务。比如这个系统服务的是账号定位、素材分析、客户需求还是产品机会？什么信息算高价值？什么关系只是弱信号，什么关系才有强证据？先有这些业务标准，后面的数据分析和长期沉淀才有意义。

## 参考了哪些方向

这个想法不是孤立的，GitHub 上已经有很多相近方向。我们可以借鉴它们的好东西，但不必照搬它们的复杂度。

- [Trace2Skill](https://github.com/Qwen-Applications/Trace2Skill)：值得借鉴“从执行轨迹里提炼可迁移 skill”。也就是说，不只看最终答案，还要看中间做错了什么、怎么修正、哪些路径有效。
- [SkillOpt](https://github.com/microsoft/SkillOpt)：值得借鉴“小步修改 + 验证门槛”。每次不要重写整个系统，只改一个提示词、规则或流程项，然后看下一次是否真的变好。
- [A-Evolve](https://github.com/A-EVO-Lab/a-evolve)：值得借鉴闭环感：执行、观察、进化、验证、重新载入。我们的版本可以更轻，但也应该有“没验证就不算学会”的原则。
- [cass-memory](https://github.com/Dicklesworthstone/cass_memory_system)：值得借鉴“procedural memory”。真正有价值的记忆不是资料堆积，而是以后怎么做事、怎么避坑、怎么判断。
- [OneBrain](https://github.com/onebrain-ai/onebrain)：值得借鉴 plain Markdown 和 local-first 的气质。先让普通文件跑起来，不急着变成平台。
- [Reflect skill](https://github.com/theprint/openclaw-reflect-skill)：证明“反思 + 记忆巩固”适合做成 skill，但我们的重点更偏公开方法论，而不是某个 agent 的内部维护流程。

所以这套方法可以吸收三个关键词：看轨迹、小步改、有验证。

最重要的约束是保持轻：不要为了沉淀而沉淀，不要把原始材料复制一遍，也不要一上来就做数据库、看板或复杂架构。先让业务标准跑起来，让数据真的能校准下一次判断。

它和完整记忆系统的区别在这里：完整系统关心怎么存、怎么检索、怎么跨工具调用；这套 skill 先关心一个更小的问题：这次数据到底让业务标准变清楚了吗？如果没有改变下一次判断，那就还只是总结，不是进化。

也就是说，它不是要替代长期记忆系统，而是给长期记忆系统提供更干净的输入：不是把所有内容都塞进去，而是先判断哪些业务规则值得沉淀，哪些数据案例值得成为下一轮判断样本。

示例提示词：

```text
Use $standard-evolver to help me describe my business, build an initial analysis standard, then analyze my data against it.
```

中文也可以这样说：

```text
用 standard-evolver 的方法，先帮我梳理业务并建立分析标准，再根据我给的数据调整这套标准，最后沉淀成以后可复用的判断规则。
```

如果要做成更友好的入口，可以让 skill 先这样引导用户：

```text
我可以先帮你建立一套业务分析标准，再用你的数据去校准它。

先告诉我：
1. 你的业务做什么，服务谁？
2. 你希望 AI 帮你判断什么？
3. 什么结果算好，什么结果算差？
4. 你手上有哪些数据、案例、素材或记录？
5. 最终标准要直接回复，还是写入某个文档？
```

也可以给用户三个模式：

- 初始化模式：根据业务描述建立第一版业务标准。
- 分析模式：按当前标准分析新数据。
- 进化模式：根据数据证据更新标准，并沉淀新规则。

这套东西适合给网友自己研究，因为它不是某个固定产品，而是一种工作协议：任何人都可以把它接到自己的笔记、项目文档、agent、代码仓库或个人知识库里。形式可以不同，但核心只有一句话：

先建立业务标准，再让每一次数据分析都成为下一次标准进化的证据。
